# 商品评论反馈检索实验平台设计文档

**日期：** 2026-04-26

## 1. 项目目标

本项目将论文《面向用户需求理解的商品评论反馈语义检索增强研究》落地为一套可复现实验平台，围绕 `键盘`、`台灯`、`耳机`、`笔记本`、`充电宝` 五个固定品类，完成以下目标：

1. 构建一个覆盖五类商品、评论总量不少于 `10000` 条的标准化商品评论数据集。
2. 实现传统检索系统，作为论文中的关键词检索基线。
3. 实现基于类别约束的商品评论反馈语义检索系统，作为论文核心方法。
4. 实现融合检索系统，用于比较关键词匹配与语义匹配结合后的效果。
5. 构建统一评测流程，输出论文实验可直接使用的结果表格与图表。

本项目的最终交付形态为 `命令行/脚本 + 实验报告图表 + 可复现实验流程`，不包含 Web 前端。

## 2. 设计范围

项目范围限定如下：

- 数据范围严格限定为五个品类：`键盘 / 台灯 / 耳机 / 笔记本 / 充电宝`
- 数据来源采用 `公开评论数据集为主，补充可复现采集脚本为辅`
- 语义检索默认采用 `在线 API / 商用 embedding`
- 预算目标控制在 `10 美元以内`
- 评测采用自建小规模人工标注查询集
- 同一数据集、同一查询集、同一评测指标用于全部检索系统对比

不在本次实现范围内的内容：

- 图形界面或 Web 系统
- 复杂的在线服务部署
- 推荐系统、摘要系统、对话式智能客服等扩展功能
- 脱离论文主题的额外商品品类扩展

## 3. 核心设计思路

本项目不拆成三个彼此独立的散项目，而是实现为一个统一实验平台，平台中包含三套可独立运行的检索系统，共享同一套数据处理、索引构建、商品聚合和评测流程。

统一平台包含以下三套检索系统：

### 3.1 传统检索系统

以商品评论作为基本检索对象，基于中文分词和 `BM25` 计算查询与评论之间的关键词相关性，再按商品进行聚合排序，输出 `商品结果 + 评论证据`。

该系统用于：

- 作为论文中的传统关键词检索基线
- 在关键词明确类查询中提供对照结果
- 为混合检索提供一部分分数来源

### 3.2 语义检索系统

系统首先识别查询中的商品类别，再去除类别词并提取需求表达，然后使用商用 embedding 将需求文本与评论文本映射为向量，在类别约束范围内使用 `FAISS` 完成 Top-K 评论语义检索，最后聚合为商品结果。

该系统用于：

- 体现论文核心创新点
- 验证“类别约束 + 语义检索”相较传统检索的优势
- 支撑全局向量检索与类别约束向量检索之间的消融实验

### 3.3 融合检索系统

在相同候选评论集合上同时计算 `BM25` 分数与向量相似度分数，进行归一化与加权融合，再按商品聚合排序。融合公式采用论文中的线性思路：

`final_score = alpha * semantic_score + (1 - alpha) * bm25_score`

该系统用于：

- 比较单一检索方式与融合检索的稳定性差异
- 支持不同查询类型下的综合效果实验
- 为论文中的增强实验提供可复现实现

## 4. 系统架构

整体架构采用共享底座 + 三套检索实现的形式，分为四层：

1. `数据层`
   负责公开数据集整理、采集脚本补充、清洗去重、品类映射、标准化存储。
2. `索引层`
   负责构建 `BM25` 索引、向量索引和商品聚合所需的数据结构。
3. `检索层`
   负责类别识别、查询处理、评论召回、分数计算、结果排序。
4. `评测与输出层`
   负责批量实验、指标计算、结果保存、图表生成。

三套系统共享数据结构、共享评测集、共享商品聚合逻辑，避免出现数据不一致、实验不可比的问题。

## 5. 数据方案

### 5.1 数据来源

采用双轨数据方案：

1. `主数据源`
   使用公开中文商品评论数据集，优先筛选可映射到五个目标品类的数据。
2. `补充数据源`
   提供一个可复现采集脚本，用于在主数据源品类分布不均或样本不足时补充评论。

### 5.2 数据目标

最终数据集目标如下：

- 评论总量不少于 `10000`
- 五个品类全部覆盖
- 各品类评论数量尽量平衡
- 每个品类下包含足够数量的商品
- 每个商品具有多条评论，便于从评论层聚合到商品层

### 5.3 标准字段

统一后的标准字段至少包括：

- `review_id`
- `product_id`
- `product_name`
- `category`
- `review_content`
- `rating`
- `source`
- `clean_text`
- `need_tags`

必要时可增加原始字段保留列，方便追踪来源与调试。

### 5.4 清洗规则

清洗流程至少包括：

- 去除空评论、乱码评论、极短无信息评论
- 去除重复评论与近重复评论
- 清理多余空白字符与无意义噪声
- 统一品类名称并映射到论文中的五个标准品类
- 尽量稳定 `product_id`，保证评论可被聚合到商品
- 保留能反映真实使用体验的评论内容

## 6. 查询处理与检索逻辑

### 6.1 查询结构假设

查询默认符合论文中的结构：`商品类别 + 用户需求表达`。

例如：

- `适合上课偷偷用的键盘`
- `宿舍晚上用会不会打扰室友的灯`
- `适合学生党买的耳机`
- `图书馆学习用合适的笔记本`
- `出门带着方便的充电宝`

### 6.2 类别识别

类别识别采用基于词典的规则匹配，具有实现简单、可解释性强的优点。类别词典覆盖五个目标品类的常见同义表达和近义表达。

若查询中未识别到类别，则系统可采用以下策略之一：

1. 语义系统回退为全局向量检索，用于实验中的消融场景。
2. 正常检索流程提示缺少类别信息，用于主实验的规范输入场景。

主实验默认优先使用规范查询，即包含明确品类词的查询。

### 6.3 需求表达提取

类别识别后，从查询中移除类别词，保留剩余文本作为需求表达输入给语义检索。这样可以减少类别词对向量相似度计算的干扰，使匹配更聚焦于用户需求。

### 6.4 商品聚合

评论是基本检索单元，但最终输出为商品结果。聚合方式采用：

- 按 `product_id` 对召回评论分组
- 对每个商品保留 Top-N 高分评论作为证据
- 使用这些评论得分的平均值或加权平均值计算商品总分
- 输出商品名称、类别、综合得分、评论证据

## 7. 索引与模型方案

### 7.1 传统检索

- 使用中文分词
- 构建评论级 `BM25` 索引
- 支持按评论得分排序
- 支持聚合为商品排序结果

### 7.2 语义检索

- 使用在线 API / 商用 embedding 作为默认向量模型
- 控制批量向量化成本，保证总预算低于 `10` 美元
- 使用 `FAISS` 构建评论向量索引
- 默认支持 `Flat`、`IVF`、`HNSW` 三类索引结构，用于效果与效率对比

### 7.3 融合检索

- 对 `BM25` 分数与向量分数做归一化
- 使用可配置权重 `alpha` 进行线性融合
- 支持在实验中调节不同权重并比较结果

## 8. 评测设计

### 8.1 查询集

构建一套小规模人工标注查询集，覆盖三种查询类型：

1. `关键词明确型`
2. `体验反馈型`
3. `场景化隐含需求型`

### 8.2 标注层级

每条查询至少标注两类答案：

- 相关评论集合
- 相关商品集合

必要时可增加“强相关 / 弱相关”层级，以支持更细粒度分析。

### 8.3 评测指标

统一输出以下指标：

- `Precision@K`
- `Recall@K`
- `MRR`
- `商品命中率@K`
- `评论证据相关率`
- `平均响应时间`

### 8.4 实验组

实验组织为三组：

1. `基础对比实验`
   比较 `BM25`、类别约束语义检索、融合检索
2. `消融实验`
   比较 `全局向量检索` 与 `类别约束向量检索`
3. `索引结构实验`
   比较 `FAISS Flat / IVF / HNSW` 的准确率与效率差异

## 11. 风险与约束

主要风险如下：

- 公开数据集中五个指定品类可能分布不均，需要补充采集或筛选策略
- 商品评论原始字段可能不统一，需额外做品类映射与商品归并
- 商用 embedding 质量较强，但需要严格控制向量化规模与重复调用次数
- 查询集人工标注规模不会太大，因此评测需要兼顾可操作性与可信度

应对策略如下：

- 优先寻找字段较完整的中文商品评论数据
- 将清洗、映射、标注、索引构建全部脚本化
- 对 embedding 调用结果进行本地缓存，避免重复计费
- 将实验查询按类型均衡设计，保证三类查询都有代表性

## 12. 实施原则

后续实现阶段遵循以下原则：

- 先完成数据底座与统一数据格式，再实现三套检索系统
- 先保证基线系统可跑通，再扩展语义系统与融合系统
- 全部结果通过统一评测脚本比较，避免手工对比
- 保证每一步都有可复现脚本与清晰输出目录

该设计文档经确认后，将进入实现计划编写阶段，并进一步拆解为按步骤执行的开发计划。
